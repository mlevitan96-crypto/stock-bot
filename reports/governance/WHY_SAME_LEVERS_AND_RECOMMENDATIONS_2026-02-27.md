# Why we keep trying the same levers (and why nothing interesting after 24h)

**Date:** 2026-02-27  
**Context:** Governance loop has been running 24+ hours; 15 runs, all REVERT; only two lever types tried.

**Note:** The recommendations in this doc are **engineering synthesis**, not a formal vote of the board/personas. For the **board and personas’ review of all droplet data** and their **additional ideas**, see **`reports/governance/BOARD_REVIEW_DROPLET_DATA_<ts>.md`** (generated by `scripts/governance/run_board_review_on_droplet_data.py`).

---

## What we’ve been collecting and trying

- **Entry lever:** Only one variant: **min_exec_score = 2.9** (sometimes 2.7 from recommender, but autopilot uses `suggested_min_exec_score` which is 2.9 when entry is “blamed”).
- **Exit lever:** Only one variant: **flow_deterioration** tweak with **strength = 0.03** (fixed `EXIT_TWEAK_STRENGTH`).
- **Loop behavior:** Alternates entry ↔ exit (even cycle = FORCE_LEVER=exit, odd = recommender). So we get exit → entry → exit → entry, but each time we do “entry” we apply the same threshold (2.9), and each time we do “exit” we apply the same strength (0.03).

So we are not “trying more” in any real sense: we have two levers and two parameter values, and we repeat them.

---

## Why we aren’t getting anything interesting

### 1. No lever variety

- **Entry:** Only threshold 2.9 (or 2.7) is used. We never try 2.7, 2.9, and 3.0 in a planned way, or try multiple exit strengths.
- **Exit:** Only one strength (0.03). We never try 0.02 or 0.05.
- **Down-weight worst signal:** Implemented but **never selected** because `signal_effectiveness.json` on the droplet is **empty** (`{}`). So `top5_harmful` is always empty and the recommender never sets `entry_lever_type=down_weight_signal`.

So the loop is structurally stuck with “entry 2.9” vs “exit 0.03” and nothing else.

### 2. Why `signal_effectiveness` is empty

- Effectiveness is built from **joined** closed trades (attribution + exit_attribution).  
- **Signal** effectiveness needs **entry_attribution_components** on each joined row (per-signal contributions).
- The loader takes `entry_attribution_components` from **entry** records in `logs/attribution.jsonl` (field `context.attribution_components`).
- So either:
  - **Attribution writes:** Entry records in `logs/attribution.jsonl` don’t have `context.attribution_components`, or  
  - **Join:** The join (trade_id or symbol+entry_ts) doesn’t match, so joined rows don’t get entry data, or  
  - **Format:** Component format doesn’t match what `build_signal_effectiveness` expects (`signal_id` / `name`, `contribution_to_score`).

Until this is fixed, we cannot rank “harmful” vs “helpful” signals and the **down-weight worst signal** lever will never run.

### 3. Stagnation / replay never triggers

- Stagnation logic uses **expectancy_history** from **LOCK** cycles.
- We have **zero LOCKs** (all REVERT), so expectancy_history never gets a single LOCK expectancy.
- So “last 3 LOCK expectancies flat” is never true and the **replay campaign** (and replay-driven overlay) never runs.
- Result: no extra variety from replay.

### 4. Same levers → same outcomes

- Raising threshold to 2.9 has repeatedly produced **worse** expectancy (e.g. −0.22 vs baseline −0.09).
- Exit tweak has also produced worse expectancy.
- So we’re not “discovering” anything; we’re reconfirming that these two single choices don’t help. Without trying 2.7, 3.0, or other exit strengths, we can’t see if any variant would help.

---

## Recommendations

### A. Add lever variety (implemented)

- **Entry:** Rotate **min_exec_score** over **2.7, 2.9, 3.0** by cycle (`cycle % 3`) in `run_equity_governance_loop_on_droplet.sh`; exported as `GOVERNANCE_ENTRY_THRESHOLD`; autopilot uses it when building entry overlay.
- **Exit:** Rotate **strength** over **0.02, 0.03, 0.05** by cycle; exported as `GOVERNANCE_EXIT_STRENGTH`; autopilot uses it for exit overlay.
- Effect: We now try 2.7, 2.9, 3.0 for entry and 0.02, 0.03, 0.05 for exit in rotation instead of always 2.9 and 0.03.

### B. Fix attribution so `signal_effectiveness` populates

- **Verify** on droplet: do entry records in `logs/attribution.jsonl` have `context.attribution_components` (list of `{signal_id, contribution_to_score}`)?
- **Verify** join: does joined output have non-null `entry_attribution_components` for most rows?
- **Fix** the write path if needed so every entry attribution that is written includes `attribution_components` from the composite (e.g. from `_composite_meta` or equivalent). Then re-run effectiveness so `signal_effectiveness.json` is non-empty and the **down-weight worst signal** lever can be selected.

### C. After N consecutive REVERTs, force a different lever

- Option 1: After e.g. 4–6 consecutive REVERTs, force **replay campaign** (even without LOCK history) to get a replay-driven overlay and try something different.
- Option 2: After N REVERTs, run one cycle with **no overlay** (baseline) to refresh baseline metrics, then continue with rotated levers.
- This avoids “infinite repeat” of the same two levers when nothing passes.

### D. Use exit_effectiveness for exit levers (future)

- We have **exit_effectiveness** (per exit_reason_code: giveback, frequency, etc.). We could recommend “increase weight for exit reason X” or “tighten exit reason Y” instead of only varying flow_deterioration strength.
- That would add real exit-side variety instead of a single global strength.

### E. Optional: track “tried” in state

- In `equity_governance_loop_state.json`, store e.g. `tried_entry_thresholds: [2.9, 2.9, ...]` and `tried_exit_strengths: [0.03, 0.03, ...]`.
- Recommendation / autopilot could prefer the **next** untried threshold or strength when building the overlay. That gives explicit coverage instead of relying only on rotation.

---

## Summary

| Issue | Cause | Recommendation |
|-------|--------|----------------|
| Same levers over and over | Only one entry threshold (2.9) and one exit strength (0.03) used | Rotate entry 2.7/2.9/3.0 and exit 0.02/0.03/0.05 by cycle |
| Down-weight never tried | `signal_effectiveness.json` empty → no `top5_harmful` | Fix attribution so entry records have `attribution_components`; verify join |
| Nothing interesting | Same two levers → same bad outcomes; no exploration | Add variety (above) + optional replay after N REVERTs |
| Replay never runs | Stagnation uses LOCK history; we have 0 LOCKs | After N REVERTs, optionally force replay or “no overlay” cycle |

Implementing **lever variety (rotation)** gives the loop something new to try every few cycles without waiting for attribution or replay. Fixing **attribution** unlocks the down-weight lever and per-signal insight. Together, we stop repeating the same two levers and start exploring the parameter space.
