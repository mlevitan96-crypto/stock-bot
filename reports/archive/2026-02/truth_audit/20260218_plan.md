# Unified Truth Audit — Plan (adversarial design)

**Date:** 2026-02-18  
**Goal:** Single PASS/FAIL: "Is the system producing truthful, meaningful scores and decisions end-to-end?"  
**Principle:** Assume the system is lying until proven otherwise.

---

## What could lie? (by axis)

### AXIS 1 — Live vs Diagnostic Parity
- **Lie:** Live process uses a different code path or config than the diagnostic (e.g. different weights, different cache, regime). Scores in production diverge from what we get when we run the same scorer offline.
- **Detection:** Run scoring via diagnostic (same cache, same `compute_composite_score_v2`). Compare symbol set and composite scores to any logged live scores (e.g. from blocked_trades, gate events, or run.jsonl). If we cannot get live scores from logs, we verify: diagnostic uses same module and cache path as main; no branching that would make live differ.
- **FAIL:** Different symbols scored, or same symbol with meaningfully different score (e.g. >0.5 delta) without explanation.

### AXIS 2 — Signal Contribution
- **Lie:** A high-impact signal (e.g. flow, options_flow) is dead, muted, or zeroed so it never moves the needle; composite is driven by a subset of signals only.
- **Detection:** For each signal: executed (present in components), non-zero (value audit), weighted (weight > ε), contributes > ε to composite (contribution share). Flag high-impact signals with ~0 contribution.
- **FAIL:** Any high-impact signal contributes ~0 across samples.

### AXIS 3 — Score Distribution Sanity
- **Lie:** Composite scores are compressed near the floor (e.g. all in [0, 1]) or zero-heavy so gates never pass or always pass.
- **Detection:** Histogram/summary of composite scores; % below 2, below 3; mean. Compare to historical if available. Flag unnaturally narrow or zero-heavy.
- **FAIL:** Unnaturally narrow (e.g. >80% in one bucket) or zero-heavy or mean < 1.0 with no justification.

### AXIS 4 — Gate Alignment
- **Lie:** Different gates use different score semantics (e.g. one uses raw score, another composite; or threshold is applied to a different quantity). Result: inconsistent pass/fail for the same candidate.
- **Detection:** For each gate (score gate, expectancy gate, capacity, etc.): which score variable is used, which threshold, pass/fail counts from gate.jsonl. Verify expectancy uses `composite_exec_score`; MIN_EXEC_SCORE applied to same score.
- **FAIL:** Different gates use different score semantics or wrong score (e.g. expectancy still using pre-fix score).

### AXIS 5 — Entry/Exit Symmetry
- **Lie:** Entry path logs fields that exit path does not (or vice versa); join keys (e.g. trade_id) missing or unstable; attribution vs exit_attribution asymmetry so we cannot join or reason end-to-end.
- **Detection:** Sample records from attribution.jsonl and exit_attribution.jsonl; check for entry_score, exit_score, trade_id, and required keys; verify join keys exist on both sides.
- **FAIL:** Asymmetry (missing fields or join keys) that breaks join or interpretation.

### AXIS 6 — Data Freshness & Completeness
- **Lie:** Caches are stale or missing so scoring uses empty or old data; timestamps wrong (e.g. future or very old).
- **Detection:** Required caches (uw_flow_cache, expanded_intel, etc.) present and mtime recent; system time sane; log timestamps coherent.
- **FAIL:** Stale (>24h) or missing required cache; timestamps obviously wrong.

---

## Multi-model role

- **Design:** All models used to ask "what could lie?" per axis above.
- **Results:** After audit run, all models review results and ask "what still might be wrong?" in _results.md.

---

## Verdict rule

**PASS** only if ALL six axes pass.  
**FAIL** if ANY axis fails — with explicit root cause and ranked integrity failures.

No tuning; exact fixes only.
