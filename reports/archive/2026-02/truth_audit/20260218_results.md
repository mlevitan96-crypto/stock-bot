## What still might be wrong? (multi-model)

### Adversarial
- **Diagnostic vs live:** We did not prove parity because the diagnostic did not run successfully on the droplet. So we have not ruled out that live scoring uses a different code path, cache, or config. **Fix:** Ensure scripts/signal_audit_diagnostic.py exists on droplet and data/uw_flow_cache.json is loadable; re-run Axis 1–3.
- **Gate alignment:** We only checked that composite_exec_score exists in main.py. We did not verify at runtime that the value passed into the gate is the same as the composite that passed the prior gate (we fixed that earlier; audit assumes deploy). **Residual risk:** If an old binary or cached .pyc is used, gate could still use wrong score.
- **Entry/exit join:** attribution has trade_id; exit_attribution sample keys did not list trade_id (had entry_ts, exit_ts, symbol). If join is on trade_id, exit_attribution must have trade_id in production; confirm schema.

- **Signal contribution:** High-impact signals (flow, dark_pool, insider) were not audited; could still be dead or muted. Re-run Axis 2 after diagnostic produces contribution data.

### Product
- **Interpretability:** Verdict and ranked failures are clear. Exact fix is: get diagnostic running on droplet with valid cache, then re-run audit. Axis 4–6 give confidence on gate alignment, attribution/exit symmetry, and data freshness; Axes 1–3 need a successful diagnostic run to pass.
